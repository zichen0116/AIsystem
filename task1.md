# 第一阶段综合设计与需求说明（给 ClaudeCode）

本文档只用于帮助你快速对齐第一阶段 MVP 的目标、边界、优先级和注意事项，不替代后续详细设计文档与实现文档。

## 1. 项目定位

- 本任务不是简单复制 OpenMAIC，而是抽取其中适合当前项目的能力，重构为教师端可用的“课堂预演 / 教学模拟”模块。
- 当前项目技术栈是 Vue + FastAPI，OpenMAIC 是 Next.js 体系，前端不能直接照搬，页面需要用 Vue 重写。
- 后端若有可复用逻辑，优先复用；不能直接复用时，再参考源实现用 Python 重写或做适配。
- 产品定位必须站稳教师端，不开放学生端能力，不要把需求做偏成“学生学习助手”。

## 2. 第一阶段的核心目标

- 先完成“课堂预演 MVP”。
- 核心能力只保留：讲解、聚焦/高亮、翻页、可选语音。
- 预演结果必须保存成独立会话，支持后续查看、恢复、续播。
- 预演应是“可播放、可控制、可保存、可回放、可续播”的系统，而不是普通聊天页。
- 底层建议采用“时间轴 / 动作序列驱动”，聊天最多只是生成动作的方式，不应成为主承载模型。

## 3. 第一阶段明确要做的内容

- 定义独立的预演数据模型、动作协议、播放状态模型。
- 设计预演会话表与状态流转，不要直接把 `courseware` 当作预演主表。
- 完成后端预演相关接口。
- 完成 Vue 版预演播放页 MVP。
- 打通两个入口：现有 PPT / 教案生成完成后的预演引导入口，以及“我的课程”中的预演入口。
- 补齐会话保存、历史查看、恢复播放等基础链路。

## 4. 第一阶段明确不要做的内容

- 不做白板。
- 不做圆桌 / 讨论。
- 不做学生端。
- 不做用户自主上传 PPT 和对应解析、预演链路。
- 不做复杂的产品化页面打磨与大范围 UI 扩展。
- 不把预演主流程建立在通用聊天接口之上。

## 5. 对 ClaudeCode 的关键实现提醒

- 参考 OpenMAIC 时，重点看“讲解模式 + 聚焦/高亮 + 翻页 + 播放控制”这条主链路。
- 前端是本项目难点，尤其要提前验证聚焦遮罩、播放状态同步、页切换、会话恢复后的状态还原。
- 即使语音在第一阶段可选，也不要让语音阻塞 MVP 主链路；可以先做可插拔或预留式设计，完整语音增强放后续阶段。
- 所有动作在进入正式协议前，都要先确认能否存储、回放、刷新恢复、异常续播。
- 不要过早为了第三阶段需求提前塞入大量上传入口；文件存储统一改造应在后续阶段按计划推进。
- 对上传 PPT 的未来能力要控制预期，只考虑页级预演，不承诺 PowerPoint 元素级还原、复杂动画还原、图表对象精确识别。

## 6. 产品与页面侧约束

- 保留“我的课程”相关入口与承接页面。
- 删除或不接入 OpenMAIC 中偏“精品课程”的对应入口和逻辑。
- 预演页的设计目标是“播放页 / 演示页”，不是聊天页。
- 当前系统已具备 PPT 生成和教案生成能力，所以第一阶段要考虑与现有生成完成页形成自然跳转。

## 7. 需要主动和用户沟通确认的点

- 某些 OpenMAIC 前端效果是否值得完全复刻，还是先做教师场景下的等价实现。
- 如果你认为时间轴模型、会话模型、前端承载方式有更优方案，可以提出，但要先沟通后再大改。

## 8. 建议推进顺序

1. 先定义预演数据模型与动作协议。
2. 再设计会话表与状态流转。
3. 再完成后端预演接口。
4. 再完成 Vue 播放页 MVP。
5. 再接入“生成完成后入口”和“我的课程入口”。
6. 最后补齐会话保存、历史查看、恢复能力。

## 9. 最终对齐结论

- 第一阶段目标是“先把可运行、可保存、可回放的课堂预演主链路做出来”。
- 不追求一步到位覆盖 OpenMAIC 全能力。
- 优先保证架构方向正确、主链路稳定、教师端场景不跑偏。
