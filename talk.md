- **缺乏多模态临时文件的处理逻辑**

  在对话检索流的第二步中，对 temp_files 的处理逻辑写的是“读取文件内容 -> 截断为单文件最多 3000 tokens”。这种纯文本的截断逻辑无法处理图片或视频流，而赛题明确要求支持多模态解析。

  如果教师上传了一张参考图片或一段教学视频截取帧作为临时文件，纯文本读取会直接报错。对于非文本的临时文件，系统不能走 token 截断逻辑，而是需要将其转换为 Base64 格式，直接组装入支持视觉能力的大模型（如 GPT-4o 或 DeepSeek-VL）的原始 Prompt 结构中，让视觉大模型去理解图像内容。

- **级联删除的数据一致性风险**

  在删除知识库流中，执行顺序依次是校验权限、删除 ChromaDB 向量、删除本地文件、最后删除数据库。

  这在分布式或多组件系统中存在部分失败的风险。如果本地文件删除成功了，但在执行数据库 DELETE 时因为网络波动或数据库锁导致报错，那么就会产生“脏数据”：数据库里依然显示这个知识库存在，但底层的向量和文件已经灰飞烟灭。下次用户点击该库时系统就会全面崩溃。

  建议在代码实现时，将步骤 4 的数据库删除操作用事务包裹起来，或者采用软删除：先在数据库将知识库状态标记为 is_deleted = true，立刻给前端返回成功，然后把清理 ChromaDB 和本地文件的任务交给 Celery 后台异步慢慢销毁。
