# PPT Outline Card Design

更新时间：2026-03-18

## 1. Goal

把当前“LLM 直接输出 markdown 大纲 + 前端直接渲染”的流程，改成“温柔的逐轮澄清 -> 结构化大纲卡片 -> 用户选图/编辑 -> 运行时转 markdown 调用文多多生成 PPT”。

这次设计的重点不是换页面，而是让聊天记录、大纲卡片、图片选择、再次打开会话时的回显都稳定一致。

## 2. User Flow

### 2.1 输入主题后的澄清

- 用户输入主题后，不要一次性抛出多条问题
- LLM 需要温柔、自然、一条一条地询问真实教学意图
- 每次只问一个问题，等用户回答后再问下一条
- 澄清信息至少覆盖：
  - 受众/学段
  - 教学目标
  - 预计时长或页数
  - 重点展开内容
  - 风格或素材偏好

### 2.2 澄清完成后的回复

澄清完成后，助手先发送一条普通文本消息：

`明白了！我已经为你生成了大纲，请查看：`

随后再发送一条 `outline` 类型消息，对应结构化大纲卡片。

### 2.3 大纲卡片

大纲卡片按照原型显示，不直接展示原始 markdown 文本。

卡片包含：

- 主题
- 章节列表
- 每个页面（对应 markdown 的 `### 页面标题`）
- 每个页面下的文本内容
- 每个页面 2 张候选 Unsplash 图片
- 用户可在 2 张图中选 1 张，也可以不选

### 2.4 用户操作

卡片支持：

- 重新生成
- 编辑
- 确认生成 PPT

编辑规则：

- 图片 URL 不允许手工编辑
- 除图片外，其余文本都可修改
- 包括主题、章节名、页面标题、段落标题、段落内容

图片规则：

- 每个页面固定展示 2 张候选图
- 用户可选其中 1 张
- 用户也可明确设置为“不选图”

### 2.5 生成 PPT

当用户点击“确认生成PPT”时：

- 使用当前卡片中的结构化内容
- 结合用户在每页的最终选图结果
- 转换成符合文多多规范的 markdown
- 将 markdown 作为 `outlineContentMarkdown` 提交给 PPT 生成接口

## 3. Data Model

当前 `PptOutline` 只保存：

- `content`
- `image_urls`

这不足以表达卡片结构和每页候选图/已选图状态。

### 3.1 新的持久化结构

`PptOutline` 需要新增一个结构化字段，例如 `outline_payload`，用于保存卡片真源数据。

建议结构：

```json
{
  "title": "中国传统文化",
  "clarification": {
    "audience": "高中生",
    "goal": "理解传统文化核心价值",
    "duration": "20-25页",
    "focus": "哲学、节日",
    "style": "稳重、课堂教学"
  },
  "sections": [
    {
      "id": "section-1",
      "title": "章节一",
      "pages": [
        {
          "id": "page-1",
          "title": "封面：中国传统文化",
          "subtitle": "五千年文明的传承与创新",
          "blocks": [
            {
              "id": "block-1",
              "title": "段落标题一",
              "content": "段落正文"
            }
          ],
          "image_candidates": [
            { "id": "img-a", "url": "https://..." },
            { "id": "img-b", "url": "https://..." }
          ],
          "selected_image_id": "img-a"
        }
      ]
    }
  ]
}
```

### 3.2 兼容字段

短期内保留：

- `content`
- `image_urls`

用途：

- 与旧数据兼容
- 保持版本查询和已有接口在迁移期间可工作

但新的前端展示与生成链路应优先基于 `outline_payload`。

## 4. Rendering Strategy

### 4.1 聊天消息

`outline` 消息的 `metadata_` 需要能携带：

- `outline_id`
- `outline_payload`
- 候选图与已选图信息

再次打开会话时，前端从后端返回的 outline 版本数据重建出同样的大纲卡片，而不是退化为普通 markdown 文本。

### 4.2 卡片展示

前端 `OutlineCard` 改为结构化渲染：

- 顶部操作栏
- 章节/页面分组展示
- 每页的 2 张候选图
- 选中态/不选图态

编辑态也保持结构化表单，不切回“整段 markdown textarea”。

## 5. Markdown Conversion

### 5.1 转换原则

生成 PPT 前，把 `outline_payload` 转为 markdown。

映射规则：

- `#`：主题
- `##`：章节标题
- `###`：页面标题
- `####`：段落标题（可选）
- 段落正文：普通文本或列表
- 若页面存在已选图片，则插入 `![配图一](url)`
- 若页面未选图，则不插入图片行

### 5.2 文多多要求

根据现有规范：

- 一级标题 `# 主题` 必须包含且只能有一个
- 二级标题 `## 章节` 必须包含
- 三级、四级标题可按内容需要组织
- 图片必须使用标准 markdown 图片语法插入：
  - `![配图一](https://xxx.png)`
  - `![配图二](https://xxx.png)`
  - `![配图三](https://xxx.png)`
- 图片应插入在具体页面内容内部，而不是集中堆到文末
- 更准确地说，图片要跟随对应页面下的段落块输出；若某个段落块绑定了选中图片，则放在该段落文本之后哎 
- 如果用户该页选择了“不选图”，则该页不要输出任何图片 markdown 行

因此前端或后端的 markdown 转换器必须保证至少产出合规的 `# / ## / ###` 结构。

### 5.3 图片插入规范

文多多图片插入规则在本项目里按以下格式固化：

```md
# 主题

## 章节一

### 页面一

#### 段落标题一

- 段落文本内容
- 段落文本内容
![配图一](https://xxx.png)

#### 段落标题二

- 段落文本内容
![配图二](https://xxx.png)
```

实现约束：

- 图片语法必须是 `![配图X](url)`，不能改成 HTML `img` 标签
- 图片 `alt` 文案统一用“配图一 / 配图二 / 配图三”这类格式
- 同一张页面若最终只选了一张图，就只输出一条图片 markdown
- 如果后续支持一页多段分别挂图，则图片应紧跟对应段落块，而不是只跟页面标题
- 当前版本若页面只有页面级选图、没有段落级选图，则默认把图片放在该页面最后一个正文块之后

## 6. Persistence and Rehydration

### 6.1 保存时机

以下动作需要持久化 `outline_payload`：

- 初次生成大纲
- 用户编辑文本
- 用户切换图片选择
- 用户点击“不选图”
- 用户重新生成大纲

### 6.2 再次打开会话

重新进入会话时：

- 当前 outline 版本应带回完整 `outline_payload`
- 聊天区里显示的 outline message 仍然是结构化卡片
- 卡片上的文本和用户选图状态必须保持一致

## 7. Gentle Clarification Tone

LLM 在澄清阶段需要符合以下语气：

- 温柔
- 不命令式
- 不一次性列出多条问题
- 每轮只问一个问题
- 适度承接上轮用户回答

示例风格：

- “好的，我先帮你梳理一下。这个 PPT 主要是讲给什么学段的学生听呢？”
- “明白了。那这份 PPT 里，你最希望重点展开哪一部分内容呢？”
- “收到。你希望整体更偏课堂讲授，还是汇报展示一点呢？”

反例：

- 一次列出 3-5 个问题
- 语气过硬、像表单采集
- 未确认上一轮信息就跳到下一轮

## 8. Implementation Boundaries

本次需要改动：

- 后端大纲持久化结构
- 大纲生成后的消息组织方式
- 前端 outline 卡片展示与编辑方式
- 图片选择逻辑
- outline -> markdown 转换逻辑
- 会话恢复时的大纲卡片回显

本次不要求：

- 支持用户手工上传替换候选图
- 支持用户直接输入自定义图片 URL
- 在卡片内自由增删任意层级结构

## 9. Risks

### 9.1 旧 outline 数据兼容

风险：

- 历史会话只有 `content` + `image_urls`

策略：

- 前端增加降级解析：如果没有 `outline_payload`，就尝试把 markdown 解析成基础结构
- 新数据全部写入 `outline_payload`

### 9.2 markdown 与结构化数据不一致

风险：

- 如果同时把 markdown 当真源、又把卡片 JSON 当真源，容易漂移

策略：

- 结构化 `outline_payload` 作为唯一真源
- markdown 只在“生成 PPT 前”即时派生

## 10. Final Decision

最终采用：

- 结构化大纲卡片为真源
- 每页固定 2 张候选图，可选 1 张或不选
- 图片不可编辑，其他文本都可编辑
- LLM 温柔地单轮单问澄清
- 澄清完成后先发提示语，再发大纲卡片
- 生成 PPT 时由结构化数据转 markdown 调用文多多
- 重新打开聊天记录时，仍显示同样的结构化大纲卡片
