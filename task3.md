# 第三阶段综合设计与需求说明

本文档用于在 `task1.md`、`task2.md` 基础上，帮助你基于当前已落地代码完成第三阶段，并作为当前项目“课堂预演”方向的收尾需求文档使用，不替代后续实现文档与测试文档。

## 1. 当前代码现状

- 第一、二阶段主链路已经落地，当前项目已具备独立的预演会话模型、页级场景模型、生成接口、播放页、历史页、基础 TTS 链路。
- 当前前端播放器、字幕、播放控制、基础播放状态保存都已存在，第三阶段不是重做播放器，而是新增“上传文件驱动的预演输入链路”。
- 当前项目中已经存在独立上传接口、`courseware` 上传记录能力，以及 `ppt_parse_service`、`renovation_service` 等 PDF / PPT 解析基础能力。
- 当前项目技术栈仍然是 Vue + FastAPI，OpenMAIC 只能作为思路参考，不能直接搬运其 Next.js / React 页面实现。
- 当前第三阶段不再视为“中间阶段”，而是预演能力的收尾阶段，不再继续推进原计划中的第四、五阶段。

## 2. 第三阶段核心目标

- 完成“用户自主上传文件并生成预演会话”的独立闭环。
- 支持教师在预演模块内上传自己的 `PDF / PPT / PPTX` 文件，生成可播放、可保存、可回放的课堂预演会话。
- 播放时展示原文件每页渲染后的图片，尽量贴近用户原稿，不做结构化重绘。
- 当前阶段动作范围收敛为“讲解 + 翻页”，不再扩展聚焦、高亮、激光、白板、圆桌等能力。
- 上传文件预演也必须支持语音，沿用现有 TTS 配置和生成链路。
- 第三阶段完成后，应能视为当前项目课堂预演能力的阶段性收尾版本，而不是仍依赖后续阶段补齐。

## 3. 第三阶段明确要做的内容

- 在预演模块内新增独立的上传文件入口，仅服务本次预演会话。
- 支持上传 `PDF + PPT/PPTX` 两类文件。
- 对 `PPT/PPTX` 统一执行“先转 PDF，再做页级处理”的链路。
- 对文件执行页级处理，默认保持 `1页文件 = 1个预演场景`。
- 允许系统自动跳过明显空白页、封底页，但需保留跳过记录和跳过原因。
- 为每个可播放页生成原页图片、页级解析摘要、教学化讲稿、TTS 音频结果。
- 失败页尽量兜底为“保留原页图片 + 简化讲稿 / 无语音”的可播放状态，而不是让整会话失效。
- 上传后用户点击确认，进入现有预演加载页，生成完成后进入现有播放页。
- 原始上传文件、转换后的 PDF、页图、讲稿、音频及页级状态都应作为会话资产保存，支持历史页再次打开播放。

## 4. 第三阶段明确不要做的内容

- 不做统一文件存储层改造。
- 不把预演上传链路与 `courseware`、我的课程、课件中心等现有上传资产打通。
- 不支持从其他模块已上传文件直接创建预演。
- 不新增用户补充文本输入，不要求填写主题、课程名、讲解要求等表单。
- 不做聚焦、高亮、激光。
- 不做结构化重绘播放，不把上传页重新生成成系统自有 slide。
- 不再规划第四、五阶段中的白板、圆桌、更多语音控制 UI、复杂历史增强页等扩展能力。

## 5. 核心模型与架构约束

- 预演主模型继续保持为 `rehearsal_session + rehearsal_scene`，不要把 `courseware` 当作上传预演主表。
- 上传文件只是本次预演会话的输入源，不应反向侵入当前课件中心的数据结构。
- 当前第三阶段可以为上传预演单独增加“会话级资产字段”或“会话级上传来源信息”，但仍应围绕预演会话建模。
- 每个页级场景除现有播放所需字段外，应至少保留：
- 原始页码
- 是否跳过
- 跳过原因
- 页图地址
- 页级解析摘要
- 讲稿文本
- 音频状态 / 音频地址
- 失败原因 / 兜底状态
- 播放页的“总可播放页数”应基于可播放场景数，而不是原始总页数。
- 会话详情或历史信息中，应能看到原始总页数、跳过页数、失败兜底页数等摘要信息。

## 6. 文件处理与生成链路要求

- 上传文件大小上限沿用现有 `50MB`。
- 单次上传的页数上限控制在约 `30` 页，超限直接拒绝并给出明确提示，不做静默截断。
- `PDF` 直接进入页级处理链路。
- `PPT/PPTX` 必须先转为 `PDF`，再统一走页图渲染、页级解析、讲稿生成、TTS 生成流程。
- 播放展示以原页视觉为准，即每页渲染出的图片为播放主画面。
- 讲稿生成应以单页解析结果为主，并允许结合整份文件上下文做适度教学化扩写，风格更像老师现场讲解。
- 讲稿不可脱离原页内容太远，不应编造原页不存在的核心知识点。
- 翻页由系统自动控制，不要求用户上传页生成额外动作协议。

## 7. 状态设计与失败兜底要求

- 第三阶段不能要求“所有页面全部成功”后才可播放。
- 页面状态至少要区分：
- 可播放页
- 自动跳过页
- 失败兜底页
- 真正不可恢复页
- 如果页图已成功，但解析摘要、讲稿或 TTS 失败，应优先保住该页可播放性。
- TTS 失败时，页面仍可通过文本讲稿继续参与播放。
- 如讲稿主生成失败，应允许退化生成简化讲稿，而不是整页直接废弃。
- 自动跳过页不进入正常播放序列，但必须在会话里可见，并显示跳过原因，例如空白页、封底页。
- 若 `PPT/PPTX -> PDF` 转换失败，则整会话可直接失败，但错误提示必须明确指出转换失败而非泛化为“生成失败”。

## 8. 页面与交互约束

- 上传入口应位于预演模块内，不与其他备课上传入口混用。
- 用户上传文件后，不再补充任何文本输入。
- 用户点击确认后，进入当前已有的预演加载页，而不是再做一套新的任务页面。
- 加载页应尽量沿用现有“输入主题生成预演”的体验和视觉结构，只是在文案和进度说明上体现“文件解析 / 页级讲稿 / 语音生成”。
- 生成完成后进入现有播放页。
- 播放页核心保持现有播放器结构，只把当前场景内容替换为原页图片播放模式。
- 历史页中，这类上传生成的预演会话应与普通预演会话并存，但需要能体现其来源是“文件上传预演”。

## 9. 关键实现提醒

- 当前第三阶段重点不是重新造播放器，而是把上传文件产物稳定接入已有播放主链路。
- 若项目内已有 `ppt_parse_service`、`renovation_service`、现有 PDF 渲染或拆页能力，可优先复用，不要额外再造一套大型解析框架。
- 若 `PPT/PPTX` 转 `PDF` 依赖 LibreOffice 或等价运行环境，应把该依赖当作明确前置条件处理，不要隐式忽略。
- 当前阶段不应为了未来可能的聚焦、高亮等能力而引入过重的结构化元素坐标体系。
- 第三阶段只需保留足够的页级元数据，用于回放、排错、历史展示和状态追踪，不要过度设计上传资产平台。
- 播放展示既然采用原页图片，就应避免再强行把上传页转成当前 `SlideRenderer` 的文本/图片/形状元素树。
- 若现有播放引擎默认按动作序列推进，可在上传页场景下收敛为“speech + 系统翻页”的最小动作集，保持底层播放模型一致即可。

## 10. 验收口径

- 用户可以在预演模块上传 `PDF / PPT / PPTX` 并发起预演。
- 上传后直接点击确认，进入现有加载页。
- 生成完成后可进入播放页，按原页图片顺序进行讲解与翻页播放。
- 上传文件预演也具备语音能力，沿用现有 TTS 配置。
- 空白页、封底页会自动跳过，但会话里可见跳过记录和原因。
- 部分页失败时，整会话仍尽量可播放，失败页会走兜底逻辑。
- 历史页可再次打开该会话，并继续播放此前生成的内容。
- 原始文件、转换 PDF、页图、讲稿、音频等会话资产不会因页面刷新或会话关闭而丢失。

## 11. 建议推进顺序

1. 先明确上传预演会话的会话级 / 页级数据结构与状态字段。
2. 再完成 `PDF / PPT / PPTX` 的统一文件处理链路。
3. 再完成页图、页级解析摘要、讲稿、TTS 的串联生成逻辑。
4. 再把上传预演接入现有加载页和播放页。
5. 最后补齐历史回放、跳过页可见、失败兜底和错误提示。

## 12. 最终对齐结论

- 第三阶段不是做通用上传平台，而是做“上传文件驱动的课堂预演闭环”。
- 本阶段收口后的课堂预演能力，应包含两条主链路：
- 现有主题生成预演链路
- 新增上传文件生成预演链路
- 当前项目在第三阶段后不再继续规划第四、五阶段，因此本阶段要把上传预演、会话保存、历史回放、失败兜底一起做到可交付。
- 当前实现策略应始终坚持“范围收敛、主链路稳定、原页视觉优先、教师端场景不跑偏”。
